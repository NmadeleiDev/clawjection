# ClawJection v1 Draft

**Status**: Public draft  
**Last updated**: 2026-03-26  
**Purpose**: Define a minimal, portable ClawJection standard that any capable OpenClaw agent can execute from `clawjection.yaml`.

---

## 1. Overview

ClawJection is a packaging standard for turning an OpenClaw instance into a role-specific operator by applying a customization bundle.

Examples:
- project manager with Jira + `gogcli`
- founder assistant with calendar, CRM, and board-prep workflows
- growth operator with browser and analytics tooling
- support triage agent with helpdesk and incident-response setup

Core model:
- the user gives the agent a ClawJection source
- the agent reads `clawjection.yaml`
- the agent executes the bundle entrypoint
- the bundle modifies the local OpenClaw runtime
- the bundle returns structured results, including what the agent should ask the user to do next

ClawJection is an OpenClaw-native standard, not a hosted-only OpenClaw Setup feature. OpenClaw Setup may later productize it, but that is an adaptation layer, not the definition of the standard.

---

## 2. Design Goals

- Standardize the minimum contract needed for agents to install a ClawJection reliably.
- Keep the bundle format flexible: only `clawjection.yaml` is standardized.
- Make pack execution portable across OpenClaw environments.
- Let packs modify workspace files, skills, plugins, and local tooling.
- Let packs explicitly tell the agent what user follow-up is required after install.
- Keep the v1 contract small enough to implement quickly.

## 3. Non-Goals

- ClawJection does not guarantee sandboxing or safety by itself.
- ClawJection does not define how every host environment enforces security policy.
- ClawJection v1 does not standardize uninstall or rollback.
- ClawJection v1 does not require a dedicated ClawJection skill.
- ClawJection v1 does not standardize internal bundle directory names beyond `clawjection.yaml`.

---

## 4. Normative Terms

The key words **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, and **MAY** are used in the RFC sense.

---

## 5. Terminology

- **Bundle**: a ClawJection package as delivered to the agent.
- **Manifest**: the `clawjection.yaml` file at bundle root.
- **Bundle root**: the directory containing `clawjection.yaml`.
- **Entrypoint**: the executable file declared by `clawjection.yaml`.
- **Runner**: the code path that reads the manifest and invokes the entrypoint. This may be agent-driven or platform-assisted.
- **Target runtime**: the OpenClaw runtime being modified.
- **Post-install guidance**: structured instructions returned by the pack telling the agent what to ask the user after install.

---

## 6. Bundle Rules

### 6.1 Required files

A valid ClawJection bundle MUST contain:
- `clawjection.yaml`
- the file referenced by `entrypoint.path`

### 6.2 Everything else is arbitrary

Beyond `clawjection.yaml`, bundle layout is unconstrained.

A bundle MAY contain any additional files the entrypoint wants to use, including:
- scripts
- templates
- bundled skills
- plugins
- binaries
- assets
- docs

These names are not part of the protocol. They are only internal bundle structure.

### 6.3 Best practice

Authors SHOULD keep bundles readable and structured, but readability is not a protocol requirement.

---

## 7. Supported Sources

ClawJection v1 SHOULD support these source types:
- Git repository URL
- direct archive URL
- local archive
- local directory

### 7.1 Accepted archive formats

ClawJection v1 runners SHOULD support:
- `.zip`
- `.tar.gz`
- `.tgz`

### 7.2 Bundle root detection

After fetch/unpack, the runner MUST identify one bundle root.

The bundle root is valid only if:
- it contains `clawjection.yaml`
- that manifest parses successfully

If an archive expands to a single top-level directory containing `clawjection.yaml`, that directory is the bundle root.

If an archive expands to multiple top-level entries, the runner MAY reject it unless exactly one valid bundle root can be identified safely.

### 7.3 Redirects and nested archives

- Remote redirects MAY be followed by the runner, subject to host policy.
- Nested archives inside the bundle are not part of the standard and SHOULD NOT be auto-expanded by default.

### 7.4 Integrity

ClawJection v1 MAY include optional provenance or checksums, but does not require them.

Hosts such as OpenClaw Setup MAY require stronger rules such as commit pinning or checksum verification.

## 8. Manifest Schema

### 8.1 Required fields

The manifest MUST contain:

```yaml
schema_version: 1
name: Jira Project Manager
version: 0.1.0
entrypoint:
  path: main.py
```

Required top-level fields:
- `schema_version`
- `name`
- `version`
- `entrypoint`

Required `entrypoint` fields:
- `path`

### 8.2 Optional fields

Optional top-level fields:
- `slug`
- `description`
- `author`
- `source`
- `inputs`
- `compatibility`
- `capabilities`
- `defaults`

### 8.3 Field semantics

#### `schema_version`

- MUST be an integer
- v1 bundles MUST use `1`

#### `name`

- MUST be a non-empty string

#### `version`

- MUST be a non-empty string
- semantic versioning is RECOMMENDED but not required

#### `slug`

- SHOULD be stable across versions
- SHOULD be lowercase with hyphens

#### `description`

- MAY be a human-readable summary

#### `author`

If present, `author` MAY contain:
- `name`
- `url`

#### `source`

If present, `source` MAY contain source metadata such as:
- `type`
- `url`
- `repo`
- `ref`

This is descriptive metadata, not execution input.

#### `entrypoint.path`

- MUST be a relative path from bundle root
- MUST NOT be absolute
- MUST NOT escape bundle root via `..`
- symlink handling is runner-defined, but the resolved target MUST remain inside bundle root

#### `inputs`

If present, `inputs` MUST be an array of input definitions.

Each input definition MAY contain:
- `key`
- `label`
- `description`
- `type`
- `required`
- `default`
- `enum`
- `secret`

Required fields for each input:
- `key`
- `type`

Allowed input types in v1:
- `string`
- `boolean`
- `enum`
- `url`
- `secret`

#### `compatibility`

If present, `compatibility` MAY contain:
- `openclaw.min_version`
- `openclaw.max_version`
- `runtime.os`
- `runtime.arch`
- `runtime.requires`

This is declarative compatibility metadata. Hosts MAY enforce it.

#### `capabilities`

If present, `capabilities` MUST be an array of strings.

Suggested capability strings:
- `workspace.read`
- `workspace.write`
- `skills.write`
- `plugins.write`
- `packages.install`
- `network.fetch`
- `auth.prompts`

In v1, capability strings are primarily disclosure metadata unless the host chooses to enforce them.

#### `defaults`

If present, `defaults` MAY contain default input values keyed by input key.

## 9. Example Manifest

```yaml
schema_version: 1
slug: jira-project-manager
name: Jira Project Manager
version: 0.1.0
description: Configure OpenClaw as a Jira-focused project manager.
author:
  name: Example Author
  url: https://github.com/example
source:
  type: github
  repo: example/clawjection-jira-project-manager
  ref: main
entrypoint:
  path: main.py
inputs:
  - key: jira_base_url
    label: Jira Base URL
    type: url
    required: true
  - key: jira_project_key
    label: Default Jira Project Key
    type: string
    required: false
  - key: jira_api_token
    label: Jira API Token
    type: secret
    required: false
compatibility:
  openclaw:
    min_version: "2026.3.0"
  runtime:
    requires:
      - python3
capabilities:
  - workspace.write
  - skills.write
  - packages.install
  - auth.prompts
```

## 10. Execution Contract

### 10.1 Bundle root

The runner MUST execute the bundle from the bundle root.

### 10.2 Invocation

ClawJection v1 defines one required lifecycle action:
- `apply`

The runner MUST invoke the entrypoint for `apply`.

Optional lifecycle actions:
- `inspect`
- `validate`

Runners MAY support them. Bundles MAY implement them. They are not required in v1.

### 10.3 Entrypoint arguments

The runner MUST invoke the entrypoint with the lifecycle action as the first argument.

Examples:

```bash
python main.py apply
bash run.sh apply
```

How the runner chooses the interpreter is host-defined. The manifest standardizes `entrypoint.path`, not interpreter selection.

### 10.4 Working directory

The runner MUST set the current working directory to the bundle root before invoking the entrypoint.

### 10.5 Environment variables

The runner SHOULD provide these environment variables:
- `CLAWJECTION_ACTION`
- `CLAWJECTION_PACK_DIR`
- `CLAWJECTION_RESULT_PATH`
- `CLAWJECTION_INPUTS_JSON`
- `CLAWJECTION_TARGET_RUNTIME`

Optional environment variables:
- `CLAWJECTION_INSTANCE_ID`
- `CLAWJECTION_WORKSPACE_DIR`
- `CLAWJECTION_AGENT_ID`
- `CLAWJECTION_STATE_DIR`

### 10.6 OpenClaw config discovery

ClawJection bundles frequently need to discover the target OpenClaw workspace and runtime settings.

ClawJection v1 defines this default rule:

- the default OpenClaw config path is `~/.openclaw/openclaw.json`

ClawJection v1 also defines this optional override:

- the entrypoint MAY accept `--openclaw-config-path <path>`

Runner behavior:

- if the runner knows the OpenClaw config path, it SHOULD pass it via `CLAWJECTION_TARGET_RUNTIME` and MAY also pass `--openclaw-config-path`
- if the runner does not know the OpenClaw config path, the entrypoint SHOULD fall back to `~/.openclaw/openclaw.json`

Entrypoint behavior:

- an entrypoint that needs workspace discovery SHOULD read `openclaw.json`
- the workspace path SHOULD be resolved from that config instead of being hardcoded
- an entrypoint SHOULD support `--openclaw-config-path` when practical

This keeps workspace discovery tied to the actual local OpenClaw runtime configuration.

### 10.7 Inputs

The runner MUST pass resolved inputs in `CLAWJECTION_INPUTS_JSON` as a JSON object.

If an input is not provided and has no default, it MUST be absent from the JSON object unless the host has another explicit convention.

### 10.8 Exit codes

- exit code `0` means the entrypoint completed successfully enough to produce a result
- non-zero exit code means execution failure

The runner SHOULD still read `CLAWJECTION_RESULT_PATH` if it exists, even on non-zero exit.

## 11. Result Contract

### 11.1 Result file

The entrypoint SHOULD write a JSON file to the path in `CLAWJECTION_RESULT_PATH`.

If the file is absent, the runner MAY treat execution as failed or partially successful according to host policy.

### 11.2 Required result fields

If a result JSON file is written, it MUST contain:
- `status`
- `summary`

Allowed `status` values:
- `ok`
- `needs_user_action`
- `failed`

### 11.3 Optional result fields

The result JSON MAY contain:
- `changed_files`
- `installed_packages`
- `installed_skills`
- `warnings`
- `errors`
- `user_actions`
- `agent_followup`
- `artifacts`

### 11.4 Structured user handoff

This is required conceptually for ClawJection, even if the runner tolerates missing fields in v1.

If a pack needs the user to do something after install, it SHOULD return `user_actions`.

Each `user_actions` entry SHOULD contain:
- `id`
- `title`
- `instruction`
- `kind`
- `required`

Optional fields:
- `secret`
- `input_key`
- `example`
- `url`
- `notes`

Allowed `kind` values in v1:
- `provide_input`
- `authenticate`
- `visit_url`
- `run_command`
- `manual_step`

This is the key mechanism that lets a pack tell the agent what to ask the user after entrypoint completion.

Example:

```json
{
  "status": "needs_user_action",
  "summary": "Google/Jira tooling installed. Interactive auth and Jira credentials still required.",
  "user_actions": [
    {
      "id": "gog-manual-auth",
      "title": "Start gog remote auth",
      "instruction": "Run the first-step `gog auth add` command in remote/manual mode, open the printed auth URL in a local browser, complete auth there, and return the resulting auth payload or redirect value back to the agent for the second step.",
      "kind": "manual_step",
      "required": true,
      "notes": "The agent should drive a two-step gog auth handoff: first tell the user which command to run, then ask for the returned auth value and complete the second step itself."
    },
    {
      "id": "jira-api-token",
      "title": "Provide Jira API token",
      "instruction": "Ask the user for a Jira API token and store it using the runtime's preferred secret flow.",
      "kind": "provide_input",
      "required": true,
      "secret": true,
      "input_key": "jira_api_token",
      "notes": "The pack cannot finish Jira integration until this token is available."
    }
  ],
  "agent_followup": {
    "mode": "ask_user",
    "priority": "blocking",
    "instruction": "Ask the user for the Jira API token now."
  }
}
```

### 11.5 Agent follow-up block

If present, `agent_followup` SHOULD tell the agent what to do next.

Suggested fields:
- `mode`
- `priority`
- `instruction`

Suggested `mode` values:
- `ask_user`
- `retry_after_input`
- `validate_only`
- `done`

Suggested `priority` values:
- `blocking`
- `important`
- `informational`

This is the direct answer to the Jira-token case: the entrypoint and its result should explicitly instruct the agent to ask the user for the missing token.

### 11.6 Example result

```json
{
  "status": "needs_user_action",
  "summary": "Configured Jira project manager pack and installed local tooling.",
  "changed_files": [
    "workspace/IDENTITY.md",
    "workspace/AGENTS.md",
    "workspace/memory.md"
  ],
  "installed_packages": [
    "gogcli",
    "atlassian-cli"
  ],
  "installed_skills": [
    "bundle://skills/jira-acli",
    "clawhub.ai/steipete/gog"
  ],
  "warnings": [],
  "user_actions": [
    {
      "id": "gog-manual-auth",
      "title": "Start gog remote auth",
      "instruction": "Tell the user to run the first-step `gog auth add` command in remote/manual mode, open the printed auth URL, finish auth in the browser, and then send back the auth payload or redirect value needed for the second step.",
      "kind": "manual_step",
      "required": true,
      "notes": "After the user sends the returned auth value, the agent should continue the second-step gog auth flow automatically."
    },
    {
      "id": "jira-api-token",
      "title": "Provide Jira API token",
      "instruction": "Ask the user for a Jira API token.",
      "kind": "provide_input",
      "required": true,
      "secret": true,
      "input_key": "jira_api_token"
    },
    {
      "id": "jira-auth-url",

      "required": false,

    "mode": "ask_user",

## 12. Path and Filesystem Rules

- All manifest paths are relative to bundle root.
- A runner MUST reject manifest paths that are absolute.
- A runner MUST reject manifest paths that escape bundle root via `..`.
- A runner SHOULD treat unresolved symlink behavior conservatively.

---

## 13. Security and Trust Model

ClawJection is powerful by design. It does not itself make packs safe.

Protocol-level statements:
- a pack MAY execute arbitrary logic through its entrypoint
- a pack MAY modify the target runtime
- a pack MAY request follow-up user actions

Environment-level statements:
- hosts MAY sandbox pack execution
- hosts MAY require capability review or confirmation before execution
- hosts MAY enforce compatibility rules
- hosts MAY reject remote sources
- hosts MAY block dangerous filesystem or network behavior

For OpenClaw Setup specifically:
- packs MUST run inside the hosted instance runtime, not on the control-plane host
- packs MUST NOT receive platform credentials

---

## 14. Idempotency

ClawJection v1 does not require hard idempotency, but packs SHOULD be idempotent.

Recommended expectation:
- running `apply` twice SHOULD converge on the same state
- repeated runs SHOULD NOT duplicate appended instructions or corrupt files

---

## 15. Compatibility

Compatibility metadata is optional, but packs SHOULD declare it when relevant.

Useful compatibility checks:
- OpenClaw version range
- required runtime tools such as `python3`, `bash`, `bun`, `git`
- OS or architecture assumptions

Hosts MAY enforce compatibility rules before running a pack.

---

## 16. Out of Scope for V1

These are explicitly not standardized in v1:
- uninstall flow
- rollback semantics
- upgrade semantics
- registry/discovery API
- signature verification
- strict capability enforcement
- standardized interpreter selection
- multi-step transactional execution

---

## 17. Optional ClawJection Skill

ClawJection SHOULD work without a dedicated skill.

The baseline contract is:
- the agent reads `clawjection.yaml`
- the agent fetches the bundle
- the agent runs the entrypoint
- the agent interprets structured results

An optional ClawJection skill MAY still be useful as a reference implementation for:
- safer archive handling
- more consistent confirmations
- better result interpretation
- cleaner user prompting

But the skill is not a protocol requirement.

## 19. Reference Example: Jira Project Manager

Expected behavior:
- update `IDENTITY.md` to set the project-manager role
- update `AGENTS.md` with Jira operating rules
- seed `memory.md` with process expectations
- install `gogcli`
- install Atlassian CLI
- install a bundled generic Jira ACLI skill from pack text (for example `skills/jira-acli/SKILL.md`)
- install the gog skill from ClawHub: `https://clawhub.ai/steipete/gog`
- instruct the agent to drive the two-step `gogcli` auth flow
- instruct the agent to tell the user to run the first-step `gog auth add` command, open the printed auth URL locally, approve auth, and send back the returned auth value
- instruct the agent to complete the second-step `gogcli` auth handoff after the user replies
- return a blocking `user_actions` item requesting a Jira API token
- return `agent_followup.instruction` telling the agent to complete gog auth first and then ask for the Jira API token

This example is important because it exercises the post-install handoff behavior that generic config packs often need.

### 19.1 Generic Jira Skill Text (Bundled Example)

Example bundled skill text:

```markdown
---
name: jira-acli
description: Use ACLI for Jira issues, projects, work items, JQL searches, transitions, and Atlassian CLI auth. Trigger when working with Jira operations or ACLI authentication/setup.
---

# Jira ACLI

Use this skill for Jira work via ACLI.

## Local environment

- Jira site: from `JIRA_SITE` environment variable
- Account email: from `JIRA_EMAIL` environment variable
- API token source: environment variable `JIRA_API_TOKEN`
- Local ACLI binary: `~/.local/bin/acli`

Do not write raw API tokens into files. Read from `JIRA_API_TOKEN` only.

## CLI shape

Use ACLI in this general form:

```bash
acli <product> <entity> <action> [flags]
```

For Jira, common patterns are:

```bash
acli jira auth <login|status|logout|switch>
acli jira workitem <create|edit|transition>
```

## Recommended workflow

1. Check required env vars exist: `JIRA_SITE`, `JIRA_EMAIL`, and `JIRA_API_TOKEN`.
2. Verify auth first:

```bash
~/.local/bin/acli jira auth status
```

3. If not authenticated, log in with stdin token:

```bash
printf "%s" "$JIRA_API_TOKEN" | ~/.local/bin/acli jira auth login \
  --site "$JIRA_SITE" \
  --email "$JIRA_EMAIL" \
  --token
```

4. Use `--help` on target commands before guessing flags:

```bash
~/.local/bin/acli jira --help
~/.local/bin/acli jira workitem --help
~/.local/bin/acli jira workitem create --help
```

## Getting started reference

- API token auth can be piped on stdin.
- OAuth login may be available via `acli jira auth login --web` depending on ACLI version.
- Work-item operations are typically centered on `acli jira workitem ...`.
- Prefer built-in `--help` because ACLI syntax can change across versions.

## Common examples

Create a work item:

```bash
~/.local/bin/acli jira workitem create --summary "New Task" --project "TEAM" --type "Task"
```

Edit work items by key:

```bash
~/.local/bin/acli jira workitem edit --key "KEY-1,KEY-2" --summary "New Summary"
```

Transition work items:

```bash
~/.local/bin/acli jira workitem transition --key "KEY-1,KEY-2" --status "Done"
```

## Safety notes

- Never persist raw API tokens in workspace files.
- Prefer env vars or stdin redirection for secrets.
- Confirm destructive or bulk-edit operations before running them.
```

## 20. Recommendations for OpenClaw Setup

OpenClaw Setup can adopt this standard with stricter host policy:
- curated sources only
- commit pinning
- stored execution logs
- install history and provenance
- review UI for capabilities and `user_actions`

But those are host decisions layered on top of the standard.
